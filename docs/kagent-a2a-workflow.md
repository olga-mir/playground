# Agent-to-Agent (A2A) Delegation: kagent Multi-Agent Orchestration

## Overview

This document walks through the **Agent-to-Agent (A2A) delegation** workflow in kagent — a pattern where one AI agent delegates specialized tasks to other agents, forming chains of orchestration. This enables modular, composable agent architectures where:

- **Cilium network agent** diagnoses Cilium health and delegates metric analysis to observability and PromQL agents
- **Crossplane composition fixer** in `team-charlie` queries Kubernetes resources via the `k8s-agent` in `kagent-system`
- **Namespace isolation** is enforced: team agents can only reach platform agents, and never reach other team agents

This guide covers:
1. **Architecture** — why standalone Agent CRs instead of Helm sub-charts
2. **Tool wiring** — how to declare A2A tool references
3. **Cross-namespace delegation** — RBAC and allowedNamespaces
4. **Practical validation** — step-by-step testing commands

---

## Architecture: Standalone Agents Instead of Bundled Sub-charts

### The Problem with Bundled Agents

kagent 0.9.2's Helm chart includes agents as optional sub-charts (e.g. `cilium-debug-agent`, `k8s-agent`). To customize them (add A2A tool references, change systemMessage), you'd use Helm `postRenderers` with kustomize patches:

```yaml
# ❌ This approach defeats GitOps transparency
spec:
  postRenderers:
    - kustomize:
        patchesStrategicMerge:
          - apiVersion: kagent.dev/v1alpha1
            kind: Agent
            metadata:
              name: cilium-debug-agent
            spec:
              tools: [...]
```

**Downsides:**
- Effective agent spec is invisible in git — it's generated by postRenderer
- Patches are fragile; they break if the Helm chart changes field order
- Cannot version-control agent configurations independently of Helm chart version

### The Solution: Standalone Agent CRs

Instead, we:
1. **Disable all bundled Agent sub-charts** in HelmRelease values
2. **Write our own Agent CRs directly** in the Kustomize tree (`kubernetes/namespaces/base/kagent/config/`)
3. **Keep tool servers enabled** (kmcp, grafana-mcp, kagent-tools) — only replace agents

```yaml
# ✓ Explicit, versioned, fully in git
apiVersion: kagent.dev/v1alpha1
kind: Agent
metadata:
  name: cilium-network-agent
  namespace: kagent-system
spec:
  modelConfig: claude-model-config
  systemMessage: |
    You diagnose Cilium network issues...
  a2aConfig:
    skills:
      - id: cilium-network-debug
        description: Diagnose Cilium issues and delegate metric analysis
  tools:
    - type: Agent
      agent:
        name: observability-agent
        namespace: kagent-system
    - type: Agent
      agent:
        name: promql-agent
        namespace: kagent-system
```

**Benefits:**
- Full transparency — agent spec lives in git
- Easy to modify without Helm expertise
- Version-controllable independently
- Follows GitOps principle: "all config in git"

### kagent-system (Platform Agents)

These are deployed as standalone Agent CRs and available to all tenants (via allowedNamespaces):

| Agent | Purpose | Delegates To |
|-------|---------|--------------|

| `promql-agent` | Write PromQL queries, analyse time-series data | — |
| `k8s-agent` | Inspect/manage Kubernetes resources, Crossplane XRs | — |
| `helm-agent` | Manage Helm charts and releases | — |

| Agent | Purpose | Delegates To |
|-------|---------|--------------|
| `crossplane-composition-fixer` | Troubleshoot Crossplane composition errors | k8s-agent (kagent-system) |

## Tool Wiring: How A2A References Work

### Declaring Agent Tools

An agent declares another agent as a tool using the Agent tool type:

```yaml
spec:
  tools:
    - type: Agent
      agent:
        name: observability-agent        # Agent name
        namespace: kagent-system          # Agent namespace
```

The kagent controller will:
1. Discover the target agent by name/namespace
2. Learn the target's `a2aConfig.skills` (capabilities)
3. At runtime, when the source agent calls the tool, establish an authenticated connection (using ServiceAccount credentials)

### Skills Metadata: Advertising Capabilities

Agents declare what they can do via `a2aConfig.skills`:

```yaml
spec:
  a2aConfig:
    skills:
      - id: cilium-network-debug
        description: >
          Diagnose Cilium network issues: node health, policy hit/drop counts,
          Hubble flow state. Delegates metric trend analysis to observability-agent
          and promql-agent.
        examples:
          - "Check Cilium node health and report any policy drops in the last hour"
          - "Analyse network latency trends for namespace team-alpha"
        tags:
          - cilium
          - networking
          - observability
```

Downstream agents can discover and call upstream agents by skill ID. The systemMessage guides when to delegate:

```yaml
spec:
  systemMessage: |
    You diagnose Cilium network issues: node health, policy hit/drop counts,
    Hubble flow state. For metric trend analysis delegate to observability-agent
    or promql-agent.
```

## Cross-Namespace Delegation: RBAC + allowedNamespaces

### The Challenge

By default, a ServiceAccount in namespace `A` cannot make HTTP calls to agents in namespace `B` — Kubernetes NetworkPolicy and RBAC block it. For cross-namespace A2A to work, we need:

1. **RBAC**: The source SA must have verbs (get, list, call) on the target resources
2. **allowedNamespaces**: The target agent must explicitly permit the source namespace

### Implementation: k8s-agent Example

The `k8s-agent` (in `kagent-system`) needs to allow calls from `team-charlie/crossplane-composition-fixer`. We do this in two parts:

#### 1. k8s-agent declares allowedNamespaces

```yaml
# kubernetes/namespaces/base/kagent/kagent/config/k8s-agent.yaml
apiVersion: kagent.dev/v1alpha1
kind: Agent
metadata:
  name: k8s-agent
  namespace: kagent-system
spec:
  modelConfig: claude-model-config
  systemMessage: |
    You are a Kubernetes expert assistant...
  a2aConfig:
    skills:
      - id: kubernetes-management
        description: Inspect and manage Kubernetes resources
        tags:
          - kubernetes
          - cluster-management
  allowedNamespaces:
    - kagent-system          # Same-namespace calls always allowed
    - team-charlie           # ← Allow calls from team-charlie namespace
```

#### 2. team-charlie/kagent SA gets RBAC binding

```yaml
# kubernetes/namespaces/base/kagent/kagent/config/rbac.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: kagent-crossplane-github-access
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: kagent-crossplane-github-access
subjects:
- kind: ServiceAccount
  name: kagent
  namespace: kagent-system
- kind: ServiceAccount
  name: kagent
  namespace: team-charlie     # ← Allow team-charlie/kagent to access Crossplane resources
```

With both in place:
- `team-charlie/kagent` SA has RBAC to read/write Crossplane resources
- `k8s-agent` explicitly allows calls from `team-charlie` namespace
- A2A call succeeds

## Practical Validation: Testing & Troubleshooting

### Prerequisites

Ensure your setup is complete:
- apps-dev GKE cluster provisioned and accessible
- Flux reconciliation is green across all namespaces
- HelmRelease `kagent` in `kagent-system` is Ready
- All agent pods are Running

### Check 1: Verify Agent Manifests Are Deployed

```bash
# List all Agent CRs in kagent-system
kubectl get agents -n kagent-system --context <apps-dev-context> -o wide

# Expected output:
# NAME                      READY   STATUS
# cilium-network-agent      True    Active
# k8s-agent                 True    Active
# observability-agent       True    Active
# promql-agent              True    Active
# helm-agent                True    Active
```

Verify individual agents:

```bash
# Check cilium-network-agent tool wiring
kubectl get agent cilium-network-agent -n kagent-system \
  --context <apps-dev-context> -o yaml | grep -A 10 "tools:"

# Expected: two Agent-type tools (observability-agent, promql-agent)
# - type: Agent
#   agent:
#     name: observability-agent
#     namespace: kagent-system
# - type: Agent
#   agent:
#     name: promql-agent
#     namespace: kagent-system
```

Verify cross-namespace agent:


## End-to-End Test Scenarios

### Scenario 1: Cilium Network Agent A2A Delegation

**Goal**: Verify that `cilium-network-agent` can delegate metric queries to `observability-agent` and `promql-agent`.

```bash

POD=$(kubectl get pod -n kagent-system --context <apps-dev-context> \

# - Tool responses received
# - Final response sent
```

### Scenario 2: Cross-Namespace Delegation (team-charlie → k8s-agent)

**Goal**: Verify that `crossplane-composition-fixer` can call `k8s-agent` across namespaces.

```bash

POD=$(kubectl get pod -n team-charlie --context <apps-dev-context> \
  -l app=crossplane-composition-fixer -o jsonpath='{.items[0].metadata.name}')
```

### Scenario 3: Namespace Isolation (Blocked Cross-Team Call)

**Goal**: Verify that `team-charlie` agents **cannot** reach `team-alpha` agents.

```bash
# 1. Create a minimal stub agent in team-alpha (for testing)
cat <<'EOF' | kubectl apply -f - --context <apps-dev-context>
apiVersion: kagent.dev/v1alpha1
kind: Agent
metadata:
  name: stub-agent
  namespace: team-alpha
  labels:
    openfga-baseline-test: "true"
spec:
  modelConfig: claude-model-config
  systemMessage: "Stub agent for testing isolation"
EOF

# 2. Send a prompt to crossplane-composition-fixer asking it to reach team-alpha agent
# (If the agent supports explicit tool delegation)

# 3. Monitor for rejection error
kubectl logs -n team-charlie --context <apps-dev-context> $POD | \
  grep -i "denied\|forbidden\|not allowed\|team-alpha"

# 4. Check kagent-controller logs for the rejection decision
CONTROLLER_POD=$(kubectl get pod -n kagent-system --context <apps-dev-context> \
  -l app=kagent-controller -o jsonpath='{.items[0].metadata.name}')

kubectl logs $CONTROLLER_POD -n kagent-system --context <apps-dev-context> | \
  grep -i "denied\|team-alpha\|allowedNamespaces"

# Expected:
# - Agent responds with "403 Forbidden" or similar
# - Controller logs show: "A2A delegation denied: team-charlie not in allowedNamespaces"
# - No cross-team communication occurs

# 5. Cleanup
kubectl delete agent stub-agent -n team-alpha --context <apps-dev-context>
```

## Troubleshooting A2A Issues

### Agent Pod Not Starting

```bash
# Describe the pod to see events
kubectl describe pod <agent-name-xyz> -n kagent-system --context <apps-dev-context>

# Check pod logs for errors
kubectl logs <agent-name-xyz> -n kagent-system --context <apps-dev-context>

# Common issues:
# - ModelConfig secret missing (kagent-anthropic)
# - CRD not installed (kagent.dev Agent CRD)
# - Image pull failure
```

### A2A Call Timeout

```bash
# Verify network policies allow communication
kubectl get networkpolicies -n kagent-system --context <apps-dev-context>

# Check agent connectivity
kubectl exec -it <agent-pod> -n kagent-system --context <apps-dev-context> -- \
  curl -v http://observability-agent:8083/health

# Verify tool references are correct (name and namespace match Agent spec)
kubectl get agent <agent-name> -n <namespace> --context <apps-dev-context> \
  -o jsonpath='{.spec.tools[*].agent}'
```

### RBAC Denial for Cross-Namespace Call

```bash
# Verify the calling SA has RBAC
kubectl get clusterrolebinding <role-name> --context <apps-dev-context> \
  -o yaml | grep -i subjects

# Verify allowedNamespaces on target agent
kubectl get agent <target-agent> -n <target-ns> --context <apps-dev-context> \
  -o jsonpath='{.spec.allowedNamespaces}'

# Check apiserver audit logs if available
```

## Next Steps: OpenFGA Integration (#103)

This A2A workflow establishes the **baseline for the OpenFGA authorization POC**. The standalone agents and namespace isolation:

1. Provide a stable, tested multi-agent setup
2. Establish expected behavior (who can reach whom)
3. Enable policy-driven authorization (OpenFGA tuples will replace allowedNamespaces)

Once OpenFGA is deployed, team-level access policies can be defined as tuples:
- `team-charlie/kagent` can call `k8s-agent`
- `team-charlie/kagent` cannot call `team-alpha/any-agent`
- etc.

See issue #103 for the OpenFGA implementation plan.
