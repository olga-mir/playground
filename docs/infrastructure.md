# Infrastructure & Cluster Provisioning

## Cluster architecture

Three clusters provisioned in dependency order:

1. **kind** (local) — bootstrap cluster; runs Crossplane + Flux; provisions the GKE control-plane cluster
2. **GKE control-plane** — hub cluster; runs Crossplane + Flux + platform services; provisions the apps-dev cluster
3. **GKE apps-dev** — workload cluster; runs tenant applications only (no Crossplane)

The kind cluster is purely temporary — it exists only to bootstrap GKE, then can be left running or torn down.

## Crossplane v2

Crossplane v2 removed the concept of Claims. Users now create namespace-scoped Composite Resources (XRs) directly — no separate claim CRD acting as a proxy. This simplifies authoring and aligns with Kubernetes conventions.

All GKE clusters are defined as `GKECluster` composite resources in their respective namespaces (`control-plane`, `apps-dev`).

Compositions create **infrastructure only** (GKE cluster, NodePool, connection secrets). Platform software is installed separately via Flux after the cluster is ready — this avoids circular dependencies and readiness race conditions.

## File structure

```
kubernetes/
├── clusters/           # Flux entry points — one folder per cluster type
│   ├── kind/
│   │   ├── flux-system/    # Auto-generated by flux bootstrap (do not hand-edit)
│   │   ├── platform.yaml   # → namespaces/overlays/kind
│   │   └── clusters.yaml   # → namespaces/base/gkecluster-control-plane/clusters
│   ├── control-plane/
│   │   ├── platform.yaml   # → namespaces/overlays/control-plane
│   │   └── clusters.yaml   # → namespaces/base/gkecluster-apps-dev
│   └── apps-dev/
│       └── platform.yaml   # → namespaces/overlays/apps-dev
├── namespaces/
│   ├── base/               # Namespace-scoped resources (environment-agnostic)
│   │   ├── crossplane-system/   # Crossplane install, providers, functions, compositions
│   │   ├── flux-system/         # Notification provider + alert
│   │   ├── gkecluster-control-plane/clusters/   # GKECluster XR for control-plane (namespace: control-plane)
│   │   ├── gkecluster-apps-dev/ # GKECluster XR for apps-dev (namespace: apps-dev)
│   │   ├── kagent/
│   │   ├── kagent-system/
│   │   └── kgateway-system/
│   └── overlays/           # Composes base dirs into a full cluster bill-of-materials
│       ├── kind/
│       ├── control-plane/
│       └── apps-dev/
├── components/
│   └── crossplane-compositions/  # XRDs and Compositions (base + overlays per cluster)
└── tenants/
    ├── base/               # Per-tenant Flux wiring
    └── overlays/           # Per-cluster tenant selection
```

## Provisioning flow

```
bootstrap-control-plane-cluster.sh
  1. Create kind cluster
  2. Create flux-system namespace + secrets (platform-secrets, github-webhook-token)
  3. flux bootstrap → installs Flux, pushes gotk-sync.yaml to git
  4. Patch flux-system secret with GitHub App credentials
  5. Wait for Flux to sync → Crossplane installed via Flux
  6. Wait for GKECluster XR → control-plane GKE cluster ready
  7. GitHub Actions triggered (Flux notification) → bootstraps Flux on control-plane
  8. Control-plane Flux provisions apps-dev GKE cluster (same pattern)
  9. GitHub Actions triggered → bootstraps Flux on apps-dev
```

## Working with Crossplane resources in GitOps

Direct `kubectl` edits to Crossplane composite resources will be overridden by Flux. Always:

1. **Suspend** the relevant Flux kustomization:
   ```bash
   kubectl --context kind-kind-test-cluster patch kustomization <name> -n flux-system --type=merge -p '{"spec":{"suspend":true}}'
   ```
2. Edit files locally and test
3. **Resume**:
   ```bash
   kubectl --context kind-kind-test-cluster patch kustomization <name> -n flux-system --type=merge -p '{"spec":{"suspend":false}}'
   ```
4. Commit and push — let Flux reconcile from git

## Required environment variables

Sourced from `.setup-env` before running setup scripts:

| Variable | Purpose |
|---|---|
| `PROJECT_ID` | GCP project |
| `REGION` | GCP region (e.g. `australia-southeast1`) |
| `ZONE` | GCP zone |
| `GKE_CONTROL_PLANE_CLUSTER` | Control-plane cluster name |
| `GKE_APPS_DEV_CLUSTER` | Apps-dev cluster name |
| `GKE_VPC` / `CONTROL_PLANE_SUBNET_NAME` / `APPS_DEV_SUBNET_NAME` | Networking |
| `CROSSPLANE_GSA_KEY_FILE` | Path to GCP service account key for Crossplane |
